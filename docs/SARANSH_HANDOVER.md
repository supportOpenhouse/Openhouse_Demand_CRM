## 3. The backend architecture (the part you're building)

### 3.1 Stack

| Layer | Choice | Why |
|---|---|---|
| Database | **Neon Postgres** | Akshit picked it; gives you branching, PITR, autoscale |
| API | **Vercel Serverless / Edge Functions** | Akshit picked Vercel; co-locate API + frontend deploy |
| Cache | **Upstash Redis** (or none for v1) | Hot keys: unread notif count, daily lists, current owner map |
| Queue / worker | **Vercel Cron + a long-running worker on Fly.io / Render** | Sheet sync, LSQ sync, materialized view refresh |
| Auth | **Google SSO** (NextAuth or Clerk) | All users are on `@openhouse.in` workspace |
| Realtime | **Postgres LISTEN/NOTIFY** + Server-Sent Events | For nudges + notifications without polling |

### 3.2 Architectural pattern — Event Sourcing + Materialized Projections

This is the same shape Stripe, HubSpot, and Linear use:

```
                          ┌────────────────────┐
   API write request ───► │   events table     │  ← single source of truth
                          │   (append only)    │     partitioned by month
                          └─────────┬──────────┘
                                    │
                  ┌─────────────────┼─────────────────┐
                  ▼                 ▼                 ▼
          Postgres triggers   Async workers     LISTEN/NOTIFY
              (sync)            (queued)         (realtime)
                  │                 │                 │
                  ▼                 ▼                 ▼
        ┌──────────────┐   ┌────────────────┐  ┌──────────────┐
        │ visits.      │   │ mv_broker_     │  │ websocket    │
        │ current_*    │   │ stats          │  │ to client    │
        │ notifications│   │ mv_dashboard   │  │              │
        └──────────────┘   └────────────────┘  └──────────────┘
              (hot read)     (5-min refresh)     (instant push)
```

Why this beats a plain CRUD schema:
- Audit log is automatic — no separate audit table to maintain
- New event types don't need schema migrations — `events.payload` is JSONB
- Undo / time-travel comes for free
- Reads are O(1) — projection tables are indexed; no joins or replays

### 3.4 Full DDL — every column

See [BACKEND_SCHEMA.md](BACKEND_SCHEMA.md) for the complete column-by-column schema with types, indexes, and the JSONB extension slots that let you add fields without migrations.

### 3.5 Why these specific tables — design rationale

**Why a separate `buyers` table?** The visitors sheet today has 7,691 visit rows but `lead_occurrence_count` shows many buyers revisit. Without a buyer entity, a phone correction has to update N rows and risks divergence. Salesforce, HubSpot, and Pipedrive all separate Contact from Activity.

**Why is `followups` append-only?** Every save creates a new row. The visit's "current" stage/status is derived from the latest followup (via a trigger that updates `visits.current_stage` etc.). This gives you free history, free undo, and free audit — without a separate audit log. This is the HubSpot Timeline pattern.

**Why temporal tables for assignments?** The question "who owned this CP when this nudge was sent" must always be answerable. Stripe uses this pattern for every account change. A flat `owner_id` column on brokers overwrites history; a temporal table preserves it for free.

**Why the universal `events` table?** Stripe Events API model. Every mutation writes an event row. Adding a new derived view becomes "write a new materialized view from existing events, deploy, drop old view." No migration.

**Why JSONB on most tables?** ~80% of "new feature" requests over time become "add this field" — JSONB lets you do that with zero schema changes. The HubSpot Custom Object framework is just a wide JSONB column with a schema registry.

### 3.6 Speed strategy — sub-100ms reads

| Tactic | Source | Where you apply it |
|---|---|---|
| Cursor pagination (no OFFSET) | Slack, Linear | Visit list: `WHERE created_at < $cursor LIMIT 50` |
| Covering indexes | Stripe | `idx_visits_dashboard ON visits(rm_user_id, visited_at DESC) INCLUDE (current_stage, current_status, society_name)` |
| Partial indexes | Notion | `WHERE current_stage NOT IN ('booking','ats','cancelled')` |
| Materialized views | Airbnb | `mv_broker_stats` for d30/60/90 counts; refresh every 5 min |
| Connection pooling | Heroku | Neon's built-in pgbouncer |
| GIN on JSONB | Sentry | Filtering by `intent.closing_signal` works with no migration |
| LISTEN/NOTIFY → SSE | Supabase | Push notif insert → client renders without re-fetch |

### 3.7 Scalability — 10x growth without re-architecture

| Concern | Today | 10x | Mitigation already baked in |
|---|---|---|---|
| `visits` rows | 7,691 | ~100,000 | Index on `(broker_id, visited_at DESC)` |
| `followups` rows | 0 | ~500,000 | **Monthly partitions** |
| `notifications` rows | 0 | ~2,000,000 | Monthly partitions; delete read >90 days |
| `events` rows | 0 | ~5,000,000 | Monthly partitions; cold storage after 12 months |
| Peak writes | low | ~50 w/s | Single append-only `events` log — no contention |
| Sheet sync | 2 min | 20 min | `_build_seed.py` becomes upsert, not regenerate |

**You will not need to shard until ~100x.** Single Neon instance + a read replica handles everything until then.

## 5. The LeadSquared migration — 5 phases

LSQ holds the operational source of truth for all followup data today. Migration is the hardest part.

### 5.1 What LSQ has that you need

From `leadsquared/snapshots/`:
- 142 lead fields per broker / buyer
- 23 custom activity types — the ones we use: 213 (Visit to be Scheduled), 214 (Visit Status), 215 (Negotiation), 216 (Booking Done), 217 (ATS Signed), 220 (Payment Status), 221 (After Visit Follow Up), 212 (Demand Phone Call), 218 (Inventory not available)
- 49 LSQ users (vs our 28 — bridge via name matching, then manual confirm)
- 2 webhooks already configured

### 5.2 The mapping — LSQ activities → our `followups`

| LSQ activity | Code | `followups.stage` | `followups.buyer_status` | Extra fields go to |
|---|---|---|---|---|
| Visit to be Scheduled | 213 | (creates a `visits` row) | n/a | `visits.selected_at` ← `mx_Custom_1` |
| Visit Status | 214 | derived from `mx_Custom_1` | `mx_Custom_6` | `visits.cancelled_reason` if cancelled |
| AVFU | 221 | `revisit_scheduled` if `mx_Custom_2` set, else `avfu` | `mx_Custom_3` (Pipeline Status) | `next_followup_date` ← `mx_Custom_1` |
| Negotiation | 215 | `negotiation` | `mx_Custom_4` | `metadata.cancelled_reason` ← `mx_Custom_2` |
| Booking Done | 216 | `booking` | `mx_Custom_9` | `metadata.token_amount`, `sale_amount`, `brokerage_pct` |
| ATS Signed | 217 | `ats` | derived from `mx_Custom_3` | `metadata.ats_signed_at` ← `mx_Custom_2` |
| Payment Status | 220 | (updates ATS followup metadata) | `mx_Custom_2` | `metadata.payment_followup` ← `mx_Custom_1` |
| Demand Phone Call | 212 | (creates an `engagement` row) | n/a | engagement notes |
| Inventory not available | 218 | `need_more` | `dead` | |

### 5.3 The 5 phases

**Phase A — Discovery & validation (Week 1)**
1. Get LSQ API key from Akshit (scopes: `Lead.Read`, `Activity.Read`, `User.Read`)
2. Run dry-run extractor for last 90 days of activities 213-221
3. Generate reconciliation report (AVFU activities with NULL pipeline status, visits with no AVFU, LSQ users without CRM mapping)
4. Fix data-quality gaps before proceeding

**Phase B — Historical backfill (Week 2)**
1. Paginated pull of ALL activities 213-221 from LSQ (expect ~50k-200k rows)
2. Map each per section 5.2 → insert into `followups` with `source='lsq_migration'`
3. Batch recompute `visits.current_*` projections
4. Validation: random 100-visit spot check with Akshit

**Phase C — Forward sync (Week 3) — RUN BOTH IN PARALLEL**

*Strategy 1 (primary): Polling every 15 min*
- Call LSQ `GetLeadActivities` filtered by `LastActivityModifiedDate > last_sync_at`
- Upsert into `followups` by `lsq_activity_id` (idempotent)

*Strategy 2 (secondary): Webhook*
- Configure LSQ webhook on activity create for codes 213-221
- Webhook hits `/api/lsq/webhook` → writes to `integration_log` → queues sync

Two channels = no data loss if either fails.

**Phase D — Cutover (Week 4) — stage-by-stage flip, NOT all at once**

| Stage | Day | What |
|---|---|---|
| Read-only CRM | Day 1 | Users browse CRM with LSQ-sourced data. No CRM writes. |
| Write to both | Day 2-7 | CRM follow-up save ALSO pushes to LSQ via API. LSQ = official record. |
| CRM primary, LSQ shadow | Day 8-14 | CRM is sole user-facing UI. Background job pushes to LSQ within 30s. |
| CRM-only | Day 15+ | LSQ push-back disabled (or kept for audit). |

Conflict rule during "write to both": **last-write-wins by timestamp. Conflicts logged to `integration_log` with severity=warning for human review.**

**Phase E — Decommission (Week 5)**
- Disable direct LSQ access for end users (read-only via CRM only)
- Keep LSQ API sync running 90 days as belt-and-suspenders backup
- Archive extraction scripts to `leadsquared/_archive/`

### 5.4 Edge cases to handle (already in the schema design)

| Edge case | Handling |
|---|---|
| LSQ activity arrives with unknown `lsq_lead_id` | Insert into `integration_log` with `succeeded=false`. Daily reconciliation alert. |
| CRM save → LSQ push fails (network) | Outbox in Redis; exponential backoff up to 24h; finally page admin. CRM data is never lost. |
| Same LSQ activity arrives twice | `idempotency_key` is the `lsq_activity_id`. Second insert is a no-op. |
| LSQ user doesn't map to CRM user | Create a "ghost" user with `team=Inactive`. Admin reassigns in bulk later. |
| AVFU pipeline_status is empty (`select_status`) | Default `buyer_status = 'unc'`. |
| Buyer phone differs between LSQ and visitors sheet | LSQ wins (operationally maintained). Diff logged to `integration_log`. |
| LSQ rate-limits during backfill | Chunk into 1000-record batches. Backfill takes hours, not minutes — it's one-time. |
